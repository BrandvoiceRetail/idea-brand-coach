## What the brand-coach MCP is

A single TypeScript streamable-HTTP MCP gateway, hosted beside Supabase, that re-surfaces brand-coach's existing coaching engines as agent-callable tools and **also acts as an MCP client of the canonical IV-OS marketing MCP**.

**Ownership split = DOING vs RECORDING.** Brand-coach **OWNS** the generative/strategy front — diagnostics (Trust Gap, Signature, avatar builder, KB search), conversation, export, plus the net-new asset chain (concept generation → publish-filter check → asset draft → test design). IV-OS **OWNS** the ledger (record/retrieve assets + tests) and the knowledge reads (brand canon, product truth, funnel).

Brand-coach **CONSUMES, never duplicates**: it reads brand canon / product truth from IV-OS and writes produced assets + recorded tests INTO the IV-OS ledger. It does **not** design asset-storage, test-storage, or brand-canon tools in this repo — it calls IV-OS. Owned tools wrap existing edge fns / TS services VERBATIM (Calculation Parity), keeping the Gen-3 lock untouched. This package mirrors the IV-OS sibling package (OVERVIEW + PLANNING surface-map + STATUS per-component) so the two sit side-by-side.

## Cross-cutting risks (own these before scaling)

1. **Calculation-Parity wrapping (Gen-3 lock).** Owned tools must wrap edge fns / TS services VERBATIM so MCP output is byte-identical to the app — the Gen-3 safety mechanism. Parity-critical TS (`trustGap.ts` scorer, `FieldPersistenceService` lock gate) must be called **in-process**, which is why a **TS gateway** is preferred over a Python FastMCP host (a Python host would force an extra edge-fn hop and could not call the lock gate in-process). Wrapping frozen edge fns verbatim is explicitly permitted and needs no R-015 sign-off; parity tests enforce no drift. Note: `callConsultant` is a **client-side TS method** (`ChatEdgeFunctionService.ts` :115, also in `SupabaseChatService.ts`) that invokes `idea-framework-consultant-claude` — not an edge-fn symbol — relevant when the in-process gateway decides edge-fn-direct vs TS-client reuse.
2. **Security must-fixes (ship with first write tool).** MF-1 confused-deputy `match_document_chunks` RPC (SECURITY DEFINER, trusts `match_user_id`, no caller check) → harden inside the function. MF-2 manual-edit-lock is TS-only (no DB trigger; `is_locked` in writable payload) → add a DB trigger forbidding AI-source writes from clearing `is_locked`, ship with `build_avatar`. MF-4 service-role key defeats RLS → NO service-role on the hot path. C1 `diagnostic-interpretation` is public/no-JWT → gateway MUST bind identity before invoking. MF-5 surface-specific log redaction (coaching msgs, KB chunks, Signature text, Bearers, claim codes — contract test, opaque-404 on validation errors).
3. **Cross-server auth boundary (D5) — brand-coach-side only.** IV-OS auth posture is **DECIDED** (single-tenant, no-auth, ADR 0001). The open decision is **brand-coach-side**: how the Supabase-JWT/RLS gateway authenticates server-to-server to a no-auth single-tenant IV-OS FastMCP (service-account vs token-passthrough).
